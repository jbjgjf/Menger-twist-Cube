# Level 2 block quotient solver (`level2-block-quotient`)

Implementation: [`packages/solver-core/src/algorithms/level2BlockQuotientSolver.ts`](../../packages/solver-core/src/algorithms/level2BlockQuotientSolver.ts). Registered under algorithm id `level2-block-quotient` (see [`packages/solver-core/src/algorithms/register.ts`](../../packages/solver-core/src/algorithms/register.ts)) and runs against the same `mengerPuzzleModel` as the [Level 1 quotient solver](level1-quotient-solver.md).

Like the Level 1 solver, it inspects the current `Cubie[]` state and derives moves from that state. It never replays `moveHistory`, never reverses user actions, and has no dependency on the Play app, React, or Three.js.

## The key idea: Menger self-similarity

Level 2 is a 9×9×9 Menger sponge with 400 unit cells. Group the grid into 27 possible 3×3×3 **blocks**; the Menger removal rule deletes the 6 face-center blocks and the center block, leaving **20 blocks** — 8 corner blocks and 12 edge blocks. That is exactly the shape of the Level 1 puzzle (8 corner cubies + 12 edge cubies), and each surviving block internally contains exactly the 20 cells of a Level 1 shape.

So a Level 2 state in which every block moves as a rigid unit *is* a Level 1 state, one fractal step up. This solver exploits that literally: it projects the block arrangement onto a synthetic Level 1 puzzle and feeds it to the existing, already-verified Level 1 quotient solver.

## Supported state class

The solver exactly solves every state generated by this move set (the **block-quotient generator set**):

| Generator | Count | Effect |
| --- | --- | --- |
| Scale-3 frame turns (`X/Y/Z`, 3 block layers each) | 9 targets | Permute whole blocks like a 3×3×3 |
| Depth-1 block extensions (`E1:*`) | 12 targets | Rotate one edge block in place |
| Depth-2 cell extensions (`E2:*`) | 240 targets | Roll one unit cell in place |

States produced by **single-layer (scale-1) turns** move individual cells across block boundaries ("tearing" blocks), and **depth-1.5 slab twists** scramble a block internally. Both put the state outside the block-rigid class; the solver detects this in phase 1 and reports an explicit failure explaining which region is torn and why. Those states are handled by the [slice-reduction solver](level2-slice-reduction-solver.md) (`level2-slice-reduction`), which reassembles the cells with commutator 3-cycles and uses this solver as its fast path for block-rigid inputs.

`apps/lab`'s Scramble control and the benchmarks both scramble within this declared generator set via the algorithm's `scrambleMovePool` (see below), so Scramble → Solve works end to end there. The Play app is manual play only and scrambles with its full frame set.

## Algorithm

Algorithm name: `level-2-block-quotient-reducer`. Four phases.

### Phase 1: block rigidity analysis

Partition the 400 cells by which block region their `currentPosition` falls in. For each of the 20 regions, verify:

1. **Purity** — all 20 cells in the region come from the same home block.
2. **Rigidity** — a single rotation `R` (one of the 24 orientation-preserving cube rotations, precomputed as exact integer matrices) maps every cell's home offset onto its current offset within the region.

Cell *orientations* are deliberately ignored here: independent cell rolls (depth-2 extensions) never move cells, so they cannot break rigidity — they are cleaned up in phase 4.

Each rigid region becomes one **macro cubie**: `homePosition` = home block coordinates, `currentPosition` = region coordinates, `orientation` = quaternion of `R`, type `corner` or `edge` by block position. If any region fails purity or rigidity, the solver stops with a structured failure naming the region and the obstruction.

### Phase 2: macro Level 1 solve

The 20 macro cubies are placed into a synthetic Level 1 `MengerPuzzleState` and handed to the registered `level1-quotient` algorithm unchanged — the same Reid-order `KPattern` projection, `cubing` search, calibration, and edge-roll quotient documented in [the Level 1 doc](level1-quotient-solver.md). Nothing is reimplemented.

The macro solve returns Level 1 moves, which lift 1:1 to Level 2 targets:

| Macro Level 1 move | Level 2 move |
| --- | --- |
| Frame `axis A`, layer `ℓ ∈ {-1,0,+1}`, angle `θ` | Scale-3 frame `A`, block layer centered at `3ℓ`, angle `θ` |
| Extension target `extension:d1:proot:s<slot>` (one edge cubie) | The *same target id* at Level 2 (one whole edge block) |

The depth-1 extension target ids coincide between levels by construction of the target generator, so the macro edge-roll fix addresses the whole-block extension directly.

The macro solve's edge quotient carries over exactly: `cubing` solves macro corner orientations fully (corner blocks end identity-oriented) and macro edges modulo roll about their own axis; the lifted `E1` extensions remove that roll. Because rigid transport maps an edge region's slot axis onto the destination region's slot axis, the residual macro roll is always about the home slot axis — the quotient is closed under the supported generator set.

### Phase 3: macro verification

The lifted moves are replayed on the real 400-cell state with the real engine move functions. After replay, every cell must be at its exact home position (rigidity + solved macro state implies this; it is verified, not assumed). Any discrepancy is reported as failure.

### Phase 4: cell extension normalization

Positions are now all home; only orientations can be wrong, and — within the supported class — every residual error is a roll about the cell's own extension axis: a cell roll applied at some slot is conjugated by subsequent rigid transports into a roll about the *final* slot's axis, and rolls about the same axis compose.

For each unsolved cell, the solver finds the depth-2 extension target selecting exactly that cell and tries `+90 / -90 / 180`, keeping the angle that makes the orientation exactly solved. A cell with a residual twist but no depth-2 target (block-corner cells have none) is outside the class and reported as failure. Final success is gated on `isExactlySolved` over all 400 cells.

## Invariants

- Cell identity: no cell is created or destroyed.
- Legal moves only: output moves are scale-3 frame targets and extension targets from the current generated target set, replayable by the Play reducer.
- Determinism: same input state and solver version produce the same output (the macro phase inherits the Level 1 solver's determinism).
- Honest failure: states outside the block-rigid class produce `success: false` with a structured explanation; the UI never fakes a solve.

## Complexity

- Phase 1: `O(400)` cell scan plus `≤ 24` rotation tests per region — `O(20 × 24 × 20)` integer vector comparisons.
- Phase 2: one Level 1 quotient solve over 20 macro cubies (see the Level 1 doc; bounded `cubing` subroutine).
- Phase 3: `O(moves × 400)` replay.
- Phase 4: `O(240 × 3)` single-cell trials.

Measured (M-series laptop, 20 seeded 20-move scrambles): 100% success, ~13ms average runtime, ~25 moves average solution length. 50-move scrambles: 100% success, ~20ms, ~36 moves.

## Benchmarking and the scramble generator set

`SolverAlgorithm` gained an optional `scrambleMovePool(model, state)` method: an algorithm may declare the generator set benchmarks should scramble with, so recorded success rates measure the algorithm against its documented scope. The benchmark runner and `apps/lab`'s single-run visualizer use it when present; algorithms without a pool are scrambled with the model's full legal move set as before.

This algorithm's pool is the block-quotient generator set, weighted roughly 55% scale-3 layers / 25% block extensions / 20% cell rolls so scrambles are dominated by permutation moves rather than orientation noise.

Run it from the CLI:

```
npm run bench -- --algorithm=level2-block-quotient --level=2 --count=20 --length=20
```

## UI surfaces

`apps/lab` picks the algorithm up from the registry with no special-casing and preselects it when Level 2 is chosen. The lab replays the solution on a centered 3D cube move by move with play / pause / stop / step / jump transport controls and a per-move duration slider (0.1s–1.0s); its debug log shows the solver's phase timings (rigidity analysis, macro solve, cubing attempts) live. See the [Level 1 doc's UI surfaces section](level1-quotient-solver.md#ui-surfaces) for the full list of lab features.

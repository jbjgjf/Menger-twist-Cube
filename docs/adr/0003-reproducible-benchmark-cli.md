# ADR 0003: seeded benchmark CLI writing to `research/results/`

## Status

Accepted

## Context

The only benchmarking surface used to be the Play app's `localStorage`-backed `SolverPanel`: runs against whatever scramble the user happened to apply by hand, with no way to compare two algorithms on the same input, no way to run benchmarks outside a browser (e.g. in CI), and nothing diffable across commits.

## Decision

Add a deterministic scramble generator (`createSeededRng` â€” a small LCG â€” plus `scrambleState`, which repeatedly calls `model.legalMoves`/`applyMove`) and a `runBenchmark(model, algorithm, spec)` function that scrambles from a list of integer seeds and runs the algorithm against each. Same model + algorithm + seed list always produces the same scrambles and the same move-count/success outcomes (wall-clock `runtime_ms` is the only field expected to vary between runs).

Expose this as a Node CLI (`packages/solver-core/src/cli/bench.ts`, run via `npm run bench`) that writes the full `BenchmarkRunResult` (records + summary) as JSON to `research/results/<model>_<algorithm>_L<level>_<timestamp>.json` by default, or an explicit `--out`/`--out-dir`. `research/results/` is committed to the repo, not gitignored â€” it is meant to accumulate as a comparable, diffable record of algorithm performance over time, the same way a changelog accumulates.

The CLI runs the same `cubing`-backed algorithm under plain Node (`cubing` requires Node â‰Ą 22.3, which is already the floor for this repo) â€” confirmed working end to end, including the cubing library's search, with no browser involved.

## Consequences

- `apps/lab`'s "Run benchmark" button and the CLI call the exact same `runBenchmark()` function â€” a result generated by one is directly comparable to, and importable alongside, a result generated by the other (`apps/lab` has a JSON file importer for this).
- Benchmark results are reviewable in a PR diff like any other committed artifact, instead of living only in a browser's `localStorage`.
- The CLI defaults to 20 seeds Ă— 20 scramble moves at level 1, matching what the original ad hoc manual testing roughly did, but every value is a flag (`--level`, `--seeds`/`--count`, `--length`) so a new algorithm can be benchmarked at a different scale without touching code.
- This intentionally does not include a results database, a comparison report generator, or a "fail CI if regressed" gate. Those are worth building once there are enough committed result files and enough algorithms that comparing them by eye stops scaling â€” premature now with one algorithm.
